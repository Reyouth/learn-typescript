    Webpack 为模块化而生

    # Webpack 为模块化而生

    webpack的目标是高效管理维护项目资源

    # 模块化

    为什么我们需要模块化？ 最早通过文件划分实现模块化。通过<script src="...">引入页面 弊端是全局环境，无私有环境，容易污染，且模块无互相的依赖关系，随着项目发展变大，维护困难。

    命名空间虽然解决了模块的私有环境问题，即让每个模块只暴露一个全局对象，然后将模块内容挂载到对象中

    window.moduleA = {
    method1: function () {
        console.log("moduleA#method1")
      }
}

    但是仍然无法实现模块的依赖关系，之后使用立即执行函数套壳来提供私有空间

    (
    function ($) {
        var name = "moduleA"

        function method1() {
            console.log(name = "#method1")
            $('body').animate({
                margin:'200px'
            })
          }
          window.moduleA = {
            method1:method1
          }
      }
)(jquery)

    虽然原生实现了，但是模块引入还是非常僵硬而不受控制。

    # 为什么需要webpack

    1. 现代前端开发复杂，后端渲染，前端原生，框架，开发方式，呼唤一种模块化开发的出现
    2. 需要高级特性来提升开发效率，比如ES6，TS，less
    3. 需要监听文件变化来实时体现到浏览器上，即热更新
    4. JS代码本身也需要模块化
    5. 开发完成后的压缩合并优化

    这些都可以交给webpack

    # 到底什么是webpack

    webpack是面向现代JS应用程序的静态模块打包工具

    • 静态模块 指开发阶段，可以被webpack引用的资源，即被webpack获取打包进bundle.js的资源

    webpack处理开发完成的应用程序时会在内部先构建模块的依赖图，并生成一或多个bundle.js

    # webpack的能力总结

    • 编译代码 比如开发环境中开发者使用ES6开发，为了适应浏览器解决兼容问题，webpack可以把代码编译成ES5版本

    • 整合模块 即打包，把多个模块打包生成名为bundle.js的文件，整合后性能提高，维护难度下降，可以避免浏览器频繁请求文件模块

    • 视万物为模块 通过安装各种前端loader，实现支持各种前端模块的加载打包，以此来通过代码控制所有资源文件的加载与否
