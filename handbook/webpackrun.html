    webpack的运行流程

    # webpack的运行流程

    webpack运行流程即把各个模块串联的流程

    1. 初始化：配置文件读取合并参数，初始化插件和对应环境
    2. 编译构建：从Entry发出，针对每个串行的Module调用Loader翻译（加载），并按照依赖关系递归编译
    3. 输出：编译后将Module组合成Chunk，Chunk转化为文件

    # 初始化

    配置文件默认为webpack.config.js ，功能是激活webpack加载项和插件

    
var path = require('path');
var node_modules = path.resolve(__dirname,'node_modules')//ath.resolve([…paths])里的每个参数都类似在当前目录执行一个cd操作，从左到右执行，
var pathToReact = path.resolve(node_modules,'react/dist/react.min.js')

module.exports = {
    //入口文件，模块构造的起点，每一个入口文件对应最后生成的Chunk，即最后生成的模块的带引用依赖的集合
    entry:'./path/to/my/entry/file.js',
    //文件路径指向
    resolve:{
        alias:{
            'react':pathToReact
        }
    },
    //生成文件，是模块构筑的终点。包括输出文件与输出路径
    output:{
        path:path.resolve(__dirname,'build'),
        filename:'[name].js'

    },
    //配置loader，设置文件后缀和对应的首处理loader以及备用处理loader
    module:{
        loaders:{
            {
                test:/\.js$/,
                loader:'babel',
                query:{
                    presets:['es2015','react']
                }
            }
        },
        noParse:[pathToReact]
    }
    //webpack各个插件对象，，在webpack事件流中执行对应方法
    Plugins:[
        new webpack.HotModuleReplacementPlugin()
    ]
}

    webpack将webpack.config.js的配置拷贝到options对象中，并加载用户配置的plugins

    之后开始初始化compiler编译对象，其中包含了钩子函数若干，继承自Tapable

    # 编译

    根据webpack.config.js的entry找出所有出口文件

    初始化完成后会调用Compiler的run来真正启动编译构建流程，包括了编译与输出

    • compile开始编译
    • make从入口分析模块以及模块依赖关系，创建模块对象
    • build-module
    • seal封装
    • emit把各个chunk输出到结果（除非特殊设置不然只有一个）

    # compile 编译

    构建了一个Compilation对象实例，里面实现了模块创建，收集依赖关系，分块，打包的任务操作对象

    # make 编译模块

    完成Compilation对象实例后，从Entry读取，执行_addModuleChain,即寻找并构筑模块链

    _addModuleChain(context,dependency,onModule,callback){
    const Dep = (dependency.constructor)
    const moduleFactory = this.dependencyFactories.get(Dep)

    moduleFactory.create({//工厂函数create来生成一个空的NormalModule对象
        dependencies:[dependency]
        ...
    },(err,module)=>{//回调中把module存入compilation.module中和dependecies.module中，
        ...
        const afterBuild = () =>{
            this.processModuleDependencies(module,err=>{
                if(err) return callback(err);
                callback(null,module)
            })
        }；
        this.buildModule(module,false,null,err=>{
            ...
            afterBuild()//随后执行buildModule过程。真正构建模块内容的过程
        })
    })
}
    • 工厂函数create来生成一个空的NormalModule对象
    • 回调中把module存入compilation.module中和dependecies.module中，
    • 随后执行buildModule过程。真正构建模块内容的过程

    # build module 模块编译

    build即调用配置的loaders，根据文件类型调用不同的loaders将模块加载成标准的JS模块

    loaders处理完后，就会使用acorn来解析内容，输出AST（抽象语法树）。 对AST的分析逻辑是： 遇到导入语句时便加入到依赖模块列表，同时根据导入内容对新的依赖模块进行递归分析，最终搞清楚所有模块的依赖关系 即递归步骤如下：

    • 调用 handleModuleCreate ，根据文件类型构建 module 子类
    • 调用 loader-runner 仓库的 runLoaders 转译 module 内容，通常是从各类资源类型转译为 JavaScript 文本
    • 调用 acorn 将 JS 文本解析为AST
    • 遍历 AST，触发各种钩子

    这个过程中数据流 module => ast => dependences => module，先转 AST 再从 AST 找依赖。这就要求 loaders 处理完的最后结果必须是可以被 acorn 处理的标准 JavaScript 语法，比如说对于图片，需要从图像二进制转换成类似于 export default "data:image/png;base64,xxx" 这类 base64 格式或者 export default "http://xxx" 这类 url 格式。 按这个流程递归处理，逐步解析出每个模块的内容以及 module 依赖关系，后续就可以根据这些内容打包输出

    # Webpack 编译过程会将源码解析为 AST 吗？webpack 与 babel 分别实现了什么？
    • 构建阶段会读取源码，解析为 AST 集合。
    • Webpack 读出 AST 之后仅遍历 AST 集合；babel 则对源码做等价转换
    # Webpack 编译过程中，如何识别资源对其他资源的依赖？
    • Webpack 遍历 AST 集合过程中，识别 require/ import 之类的导入语句，确定模块对其他资源的依赖关系
    # 相对于 grant、gulp 等流式构建工具，为什么 webpack 会被认为是新一代的构建工具？
    • Grant、Gulp 仅执行开发者预定义的任务流；而 webpack 则深入处理资源的内容，功能上更强大

    # 输出流程

    # seal

    原意密封、上锁，我个人理解在 webpack 语境下接近于 “将模块装进蜜罐” ，即封装模块。

    seal目的是生成Chunks，并对其进行优化，最终输出网页代码

    如何理解Chunks：在处理中的Chunks，即配置在Entry中的模块；最终生成的Chunks，即根据前面操作模块的依赖组合组成的包含模块与关系的集合，最后会将Chunks转成单独文件加入到输出

    # emit

    emit输出。 在Chunks最终生成，即主体内容生成完成，知道了“输出什么”， 会根据configs的onput来确定输出文件的路径等配置，即决定“怎么输出”

    emit作为开始生成文件前的钩子会被执行，这是修改最终文件的最后一个机会
